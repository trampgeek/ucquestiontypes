Note to would-be editors

This is a very complex question type with a lot of code replicated in the template and prototype extra fields. To edit, download the source code from python3_roboturtle in the ucquestiontypes repository on github. The functionality is broken into a dozen or so files, and a build.py program is used to generate the content for the template and prototype extra fields - never edit prototypeextra.html, template.py or assembledrobot.py directly, as they are all generated and will be overwritten.

About this question type

RoboTurtle uses the Python Turtle module to implement a variant of Reeborg's World - see https://reeborg.ca/reeborg.html. Students write code that drives the robot around a grid world to perform some task set by the question, such as moving to a target cell, following an exact path, or collecting items and depositing them at one or more locations.

Authoring RoboTurtle questions

IMPORTANT:
  1. You must check TwigAll or you'll get inexplicable error messages!
  2. You cannot (easily) enter the Answer (or the Preload) directly. You will need to Preview the question, use Ctrl+Alt+m to disable the UI, and copy and paste the answer from the preview back into the authoring form.
Use the visual world editor

Hand-writing the testcases JSON below is tedious and error-prone. There's a standalone visual editor at python3_roboturtle/editor.html in the repository - download it and open it directly in a browser (it's self-contained; no server or build step needed). It lets you click/drag to place walls, blocked cells, items, item dumps, paths and text-output expectations on a grid matching the live question's rendering, then export the finished JSON straight into the testcases parameter below. It also has Import, so you can paste an existing test case back in to tweak it. This is by far the easiest way to author or edit test cases.

Template Parameters
  • testcases - a JSON list of one or more tests, as described below. REQUIRED.
  • maxnumlines - the maximum number of (non-blank, non-comment, non-docstring) lines of code allowed in the student's answer. Optional, default 0 (no limit).
  • disabledfunctions - a list of names, from the command reference below, that the student is not allowed to use (e.g. ["build_wall", "toss"]). Optional.
  • showhelpbutton - set to false to hide the in-question "RoboTurtle Help" button. Optional, default true.

Commands available to students

All of the following are plain global functions, callable directly from student code (e.g. move(), not robot.move()).

Movement and actions
  • move() - move one cell forward. Raises an error if blocked by a wall or a blocked cell.
  • turn_left() - rotate 90 degrees counterclockwise.
  • turn_right() - rotate 90 degrees clockwise.
  • take() - pick up an item from the current cell, add it to the inventory, and return its type name (e.g. 'star'). Raises an error if there's nothing to take.
  • put() - place the least-recently-picked-up item from the inventory into the current cell, and return its type name. Raises an error if the inventory is empty.
  • toss() - place the least-recently-picked-up item from the inventory into the cell directly in front of the robot, and return its type name. Raises an error if the inventory is empty, the target cell is out of bounds, or it's blocked.
  • build_wall() - build a wall directly in front of the robot, i.e. at the edge of the current cell in the direction the robot is facing.
  • send_text(s) - display the string s in a text bar below the grid (only the most recent call's text is shown). Only usable in tests whose world_params includes expected_texts - see "Text output", below. Raises an error otherwise.
  • speed(n) - set the animation speed, from 1 (slow) to 10 (fast); 0 for fastest possible. Cosmetic only - has no effect on grading.
  • print_state() / print(...) - print diagnostic output to the on-screen output pane. Browser-only: both are silently disabled during actual grading on the Jobe server, so they're useful for a student debugging their own code in the preview, but must never be relied on as part of a test's pass/fail logic. Use send_text() for anything that needs to be checked.
State-checking boolean functions
  • at_goal() - true if the robot has satisfied the world's goal (reached the target, and/or matched the required path, and/or emptied all items into their dump(s), depending on world type).
  • front_is_clear() - true if the robot is able to move forward.
  • right_is_clear() - true if the cell to the right is clear.
  • wall_in_front() - true if there's a wall directly in front.
  • wall_on_right() - true if there's a wall to the right.
  • object_here() - true if there's an item in the current cell.
  • carries_object() - true if the inventory is non-empty.
  • is_facing_north() - true if the robot is currently facing north.
State-query functions
  • position() - the robot's current grid coordinates as an (x, y) tuple.
  • facing() - the robot's current heading as 'N', 'S', 'E', or 'W'.
  • target_cell() - the target cell's coordinates as an (x, y) tuple, or None if this world has no target (e.g. a plain PathWorld).
  • inventory() - a list of the type names of items currently in the inventory, ordered by when they were picked up (most recently picked up last).

The testcases format

The testcases template parameter must be a JSON list of one or more tests. Each test is a JSON object shaped like this (comments added here for explanation - real JSON can't contain comments, so strip them if copying this by hand; using the visual editor avoids the problem entirely):

{
  "world_class": "TargetWorld",       // one of TargetWorld, PathWorld, PickUpItemsWorld - see "Types of World" below
  "world_params": { },                // parameters specific to world_class, plus optionally cell_size and expected_texts - see below
  "grid_size": [10, 8],               // [width, height] in cells (NOT [rows, columns])
  "robot_start": {
    "position": [0, 0],
    "heading": "north"                // one of "north", "south", "east", "west"
  },
  "walls": [],                        // list of [[x1,y1],[x2,y2]] pairs, each horizontal or vertical
  "blocked_cells": [],                // list of [[x0,y0],[x1,y1]] rectangle corner pairs (bottom-left, top-right)
  "items": []                         // see below
}

Each entry in items looks like:

{
  "type": "leaf",         // an item type - see "Item types" below
  "position": [5, 7],     // a cell, or the string "inventory" to start in the robot's inventory
  "count": 3              // how many such items at that cell/in inventory. Optional, default 1
}

Two world_params entries apply regardless of world_class:

cell_size
Cell size in pixels. Optional, default 65.
expected_texts
See "Text output", below.

Text output

Any world type can require the student's code to produce a specific sequence of messages via send_text(). Add expected_texts to world_params - an ordered list of the exact strings the student's code must send, in order, e.g. ["Starting", "Done"]. This reserves a text bar below the grid (showing only the most-recently-sent message) and makes at_goal()/the pass condition additionally require send_text() to have been called with exactly that sequence. A handy pattern is to combine this with the return values of take()/put()/toss(), e.g. send_text(f"Picked up {take()}").

Item types

Available item type names (used for items, target_icon, dump icon/type, and labelled_cells): star, key, leaf, house (alias home), flag, target, box, trash.

Types of World

The world_class determines the goal the student must satisfy to pass a test.

TargetWorld

The robot must finish on a specified target cell.

World Parameters:

target
The 2-element array specifying the robot's target cell. Required.
target_icon
The item type used to mark the target cell - typically 'flag' (chequered), 'target' (dart-board), or 'house'/'home'. Optional, default 'flag'.

PathWorld

The robot must follow an exact sequence of cells from start to finish.

World Parameters:

path
The exact sequence of cells the robot must visit, as a list of path segments; each segment is a two-element list of 2-element locations (start and end of that segment). Segments must be horizontal or vertical. Required.

PickUpItemsWorld

A subclass of TargetWorld: the robot must finish at the target and collect all items in the grid (except explicitly excluded ones), optionally redepositing them at one or more dump locations.

World Parameters:

item_dumps
A list of dump specifications, each a dictionary with keys:
  • location - the dump's grid cell. Required.
  • type - the item type that must end up there, e.g. 'leaf'. Optional, default 'Any' (accepts any item type).
  • icon - the item type used to draw the dump's icon, e.g. 'trash' or 'box'. Optional, default 'trash'.
  • count - how many items are expected at the dump. Optional - if omitted, it's auto-derived as the number of matching items (of type, or all items if 'Any') placed anywhere in the world for this test, so e.g. "put all the leaves in the trash" doesn't need hand-counting them.
A dump whose type isn't 'Any' fails separately if any wrong-typed item ends up there, as well as if the count is wrong. If item_dumps is omitted entirely, it's only required that no items remain anywhere in the grid (i.e. they're all in the inventory).
item_dump (deprecated)
The old singular form: {"location": [x, y], "num_items": N}. Still recognised for backward compatibility with existing questions - it's mapped internally to a single-entry item_dumps list with type: 'Any' and icon: 'trash'. Ignored if item_dumps is also present. Don't use this for new questions - use item_dumps. (The visual editor only ever reads/writes item_dumps, converting any old item_dump on import.)
exclude_cells
A list of 2-element locations of cells whose items the robot does not need to gather (ignored by the check).

A worked example

A PickUpItemsWorld test: collect all keys and leaves, sorting keys into a box and leaves into the trash, then finish at the target - reporting progress via send_text() along the way.

{
  "testcases": [
    {
      "grid_size": [8, 6],
      "world_class": "PickUpItemsWorld",
      "world_params": {
        "target": [7, 5],
        "item_dumps": [
          {"location": [0, 5], "type": "key",  "icon": "box"},
          {"location": [7, 0], "type": "leaf", "icon": "trash"}
        ],
        "expected_texts": ["Starting", "Done"]
      },
      "robot_start": {"position": [0, 0], "heading": "east"},
      "walls": [[[4, 0], [4, 3]]],
      "blocked_cells": [[[2, 2], [3, 3]]],
      "items": [
        {"type": "key",  "position": [3, 2], "count": 2},
        {"type": "leaf", "position": [5, 4]}
      ]
    }
  ]
}
